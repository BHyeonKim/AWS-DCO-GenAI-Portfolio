# AWS DCO 인턴십 직무 이해를 위한 생성형 AI 활용 포트폴리오

## 프로젝트 소개

이 프로젝트는 공개된 DCO 직무 정보와 교육용 샘플 데이터를 바탕으로
DCO 직무의 기본 용어, 교육용 로그 분석, 장애 상황 문서화와 인수인계 흐름을 학습한 결과물입니다.

생성형 AI를 단순히 답을 생성하는 도구로 사용하지 않고,
용어 정리, 문서 구조화, 로그 분석, Python 코드 작성, 보고서 초안 작성과 결과 검증을 돕는 도구로 활용했습니다.

이 프로젝트는 실제 AWS 내부 시스템, 실제 운영 로그, 실제 Ticket 또는 실제 데이터센터 장비를 대상으로 수행한 프로젝트가 아닙니다.

## 프로젝트에서 수행한 주요 활동

- 공개 직무 정보를 바탕으로 DCO 직무 키워드와 인프라 기초 용어 정리
- 개인의 협업 강점을 DCO 직무 맥락과 연결한 협업 유형 리포트 작성
- 교육용 SOP를 절차·주의사항·확인 항목으로 구조화한 체크리스트 제작
- Python 표준 라이브러리만으로 교육용 로그를 파싱해 심각도·이벤트별 집계, WARNING/ERROR 로그 목록, 주요 이벤트(CRC_ERROR·LINK_DOWN·TICKET_ESCALATED) 요약을 자동 생성하는 스크립트 작성
- 로그 분석 스크립트에서 발생한 오탐(정상 복구 메시지가 CRC_ERROR로 잘못 분류되는 문제)을 직접 발견하고 이벤트명 기준 매칭으로 수정
- CRC Error·Link Down 사건을 시간순으로 정리하고, 확인된 사실(사실)과 가능한 원인(추정)·현재 확인 불가능한 정보(미확인)를 구분해 문서화
- 여러 파일(원본 로그, 분석 요약, 원인 분석)을 종합해 교육용 Incident Report 초안을 작성하고, 사건 시간·장비명·Ticket ID를 원본 로그와 한 줄씩 대조
- 4건의 Ticket을 사실 정리표(CSV)로 만들고, Severity뿐 아니라 서비스 영향·반복 여부·영향 범위·복구 여부를 함께 고려해 우선순위를 직접 결정
- 우선순위 판단 근거와 다음 담당자 인수인계 내용을 담은 Shift Handover Memo 작성
- Windows 환경에서 한글이 포함된 CSV 파일이 Excel에서 깨지는 문제를 발견하고 UTF-8 BOM으로 재저장해 해결
- Git으로 변경 이력을 관리하고, 작업 단위(차시)별로 범위를 좁혀 커밋·GitHub push 진행

## 생성형 AI 활용 원칙

* 생성형 AI의 답변은 최종 정답이 아니라 검토가 필요한 초안으로 사용했습니다.
* AI가 작성한 장비명, 시간, Ticket ID와 사건 내용을 원본 로그와 비교했습니다.
* 로그에서 직접 확인되는 사실과 가능한 추정을 구분했습니다.
* 로그만으로 확인할 수 없는 Root Cause는 확정하지 않았습니다.
* Ticket 우선순위는 AI가 대신 결정하지 않고, 학생이 근거를 비교하여 직접 판단했습니다.
* 실제 계정정보, 실제 장비정보, 실제 Ticket과 실제 AWS 내부정보를 AI에 입력하지 않았습니다.
* 모든 로그, 장비명, Ticket과 IP 주소는 교육용 샘플 데이터로 구성했습니다.

## 배운 점

- Severity(심각도)와 Priority(우선순위)는 다른 개념이라는 것을 체감했습니다. Ticket 4건을 비교해 보니 ERROR로 기록된 Link Down보다, WARNING 상태지만 이중 전원(PDU Feed A/B) 중 하나가 아직 복구되지 않고 영향받는 서버가 더 많은 사건을 먼저 챙겨야 하는 경우가 있었습니다.
- CRC Error 증가와 Link Down처럼 겉보기에는 원인이 뻔해 보이는 사건도, 로그만으로는 패치 코드·커넥터·포트 하드웨어 중 무엇이 진짜 원인인지 확정할 수 없다는 것을 직접 확인했습니다. 데이터센터 현장에서는 "복구되었다"와 "원인이 밝혀졌다"가 같은 말이 아니라는 것을 배웠습니다.
- Rack PDU의 이중 급전(dual-feed) 구조처럼, 한쪽 전원(Feed A)에 문제가 생겨도 다른 쪽(Feed B)이 서비스를 유지해 주는 이중화 설계가 있다는 것과, 이런 경우에도 "서비스 중단 없음"이 "정상 상태로 완전히 복구됨"을 의미하지는 않는다는 차이를 알게 되었습니다.
- Ticket이 Open → Escalated → Monitoring/Resolved로 상태가 바뀌는 흐름을 직접 다뤄보면서, 담당자가 바뀔 때마다 현재 상태와 남은 위험을 명확히 전달하는 절차(Escalation)가 왜 필요한지 이해할 수 있었습니다.
- Shift Handover Memo를 작성해 보면서, 교대 근무 환경에서는 "내가 무엇을 확인했는지"뿐 아니라 "다음 담당자가 무엇을 아직 모르는지"까지 함께 적어야 인수인계가 끊기지 않는다는 것을 배웠습니다.
- SOP 체크리스트를 분석하면서, 정해진 절차와 확인 항목이 있는 이유는 작업자의 순간적인 판단에만 의존하지 않고 누가 하더라도 같은 기준으로 점검하게 만들기 위한 것임을 이해하게 되었습니다.
- Incident Report에 "확인된 사실"과 "가능한 원인(추정)", "확인되지 않은 내용(미확인)"을 나누어 써보면서, 장애 보고서는 사건을 해결했다는 결과만 전달하는 문서가 아니라 다음 사람이 같은 판단을 반복할 수 있도록 근거를 남기는 문서라는 것을 알게 되었습니다.

## 프로젝트 범위와 제한

이 저장소는 DCO 인턴십 직무를 이해하기 위한 학습형 포트폴리오입니다.

- 실제 AWS DCO 업무 수행 능력을 증명하는 자료가 아닙니다.
- 실제 데이터센터 운영 절차나 내부 SOP를 포함하지 않습니다.
- 실제 장애 대응, 장비 교체 또는 운영 자동화를 수행하지 않습니다.
- 공개 직무 맥락과 교육용 샘플 데이터를 이용한 학습 결과만 포함합니다.
