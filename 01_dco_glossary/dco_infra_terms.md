## 1. 작성 방법

아래 용어 설명은 AI의 설명을 참고했지만, 최종 문장은 제가 이해한 말로 다시 작성합니다.

작성 기준:

1. 한 용어당 1~2문장으로 작성합니다.
2. 어려운 기술 용어는 쉬운 말로 바꿉니다.
3. 실제 내부정보, 실제 티켓 번호, 실제 장비정보, 실제 IP 주소는 포함하지 않습니다.
4. 원인을 하나로 단정하지 않습니다.
5. 헷갈리는 내용은 "아직 헷갈림"이라고 표시할 수 있습니다.

## 2. 내가 만든 DCO 인프라 용어 정의

### Rack

내 정의: 서버나 스위치 같은 장비를 세로로 쌓아서 고정해두는 표준 규격의 장(구조물)이라고 이해했습니다.

### Server

내 정의: 데이터를 저장하고 처리해서 다른 컴퓨터에 필요한 서비스를 제공해주는 컴퓨터라고 이해했습니다.

### NIC

내 정의: 컴퓨터나 서버가 네트워크에 연결될 수 있도록 해주는 통신용 카드(랜카드)라고 이해했습니다.

### ToR Switch

내 정의: 랙 맨 위쪽에 설치되어, 같은 랙 안에 있는 서버들을 네트워크로 연결해주는 스위치라고 이해했습니다.

### Link Down

내 정의: 네트워크 연결이 끊긴 상태를 나타내는 표시라고 이해했습니다. 다만 끊긴 원인은 케이블 문제일 수도, 장비 문제일 수도 있어 하나로 단정하기는 어렵다고 생각합니다.

### CRC error

내 정의: 데이터가 전송되는 과정에서 오류가 발생했음을 알려주는 신호라고 이해했습니다. 원인은 여러 가지일 수 있어 아직 헷갈림이 있습니다.

### SSD failure

내 정의: 저장장치(SSD)가 정상적으로 동작하지 않는 고장 상태를 뜻한다고 이해했습니다.

### Fan Alert

내 정의: 장비 내부 냉각 팬이 제대로 작동하지 않을 때 발생하는 경고라고 이해했습니다.

### SOP

내 정의: 특정 작업을 처리할 때 따라야 하는 정해진 절차라고 이해했습니다.

### SLA

내 정의: 서비스 처리 시간이나 품질에 대해 미리 정해둔 약속(기준)이라고 이해했습니다.

### Escalation

내 정의: 현재 담당자 선에서 해결하기 어려운 상황을 상위 담당자나 다른 담당 조직에 전달하는 절차라고 이해했습니다.

### Incident Report

내 정의: 발생한 장애나 이상 상황의 경과와 조치 내용을 기록하는 보고서라고 이해했습니다.

### Handover Memo

내 정의: 근무 교대 시 다음 근무자에게 진행 중인 작업이나 특이사항을 전달하는 인수인계 메모라고 이해했습니다.

### Temperature warning

내 정의: 장비나 랙 내부 온도가 정상 범위를 벗어났을 때 발생하는 경고라고 이해했습니다.

## 3. 헷갈리는 용어 비교

내가 비교한 용어: SOP와 SLA

내가 이해한 차이: SOP는 "어떤 순서와 방법으로 작업할지"를 정해둔 절차이고, SLA는 "얼마나 빠르게, 어느 정도 수준으로 처리해야 하는지"에 대한 기준(약속)이라고 이해했습니다. 즉 SOP는 진행 방식에 가깝고, SLA는 처리 목표치에 가깝다고 생각합니다.

아직 헷갈리는 부분: SLA 기준을 넘겼을 때 Escalation이 자동으로 이루어지는 것인지, 담당자의 판단이 별도로 필요한 것인지는 아직 헷갈립니다.

## 4. 앞으로 사용할 학습 보조 프롬프트

### 프롬프트 1. 모르는 DCO 용어 설명용

역할: DCO 인턴십 직무를 이해하려는 학생을 돕는 학습 코치

목표: 내가 모르는 DCO 용어를 비전공자도 이해할 수 있는 쉬운 말로 설명해달라고 요청

상황: 공개된 DCO 직무 설명에서 접한 용어이며, 실제 내부 절차가 아닌 교육용 맥락으로 이해하려는 상황

출력 형식: 용어명 / 쉬운 뜻(1~2문장) / 헷갈릴 수 있는 부분 표시

제한 조건: 실제 회사명·장비명·티켓 번호 등 내부정보처럼 보이는 내용 포함하지 않기, 원인을 하나로 단정하지 않기

### 프롬프트 2. 비슷한 용어 비교용

역할: DCO 인턴십 직무를 이해하려는 학생을 돕는 학습 코치

목표: 뜻이 비슷해서 헷갈리는 두 용어의 차이를 비교해달라고 요청

상황: 교육 자료에서 두 용어가 함께 등장했는데 정확한 구분 기준을 모르는 상황

출력 형식: 용어 A와 B 각각의 쉬운 뜻 / 두 용어의 핵심 차이 1~2문장 / 아직 헷갈리는 부분

제한 조건: 실제 내부정보 포함하지 않기, 확실하지 않은 차이는 단정하지 않고 "~일 수 있다" 수준으로 표현하기

### 프롬프트 3. 장애 표현 확인 순서 정리용

역할: DCO 인턴십 직무를 이해하려는 학생을 돕는 학습 코치

목표: 특정 장애 표현(예: Link Down, CRC error)이 나타났을 때, 교육용 시나리오 기준으로 확인해볼 수 있는 순서를 정리해달라고 요청

입력: 가정한 교육용 샘플 로그 상황(실제 장비명·티켓 번호 제외)

출력 형식: 상황 요약 / 확인해볼 수 있는 순서(체크리스트 형태) / 이 경험이 연결되는 협업 역량

제한 조건: 실제 장비를 점검하는 절차처럼 단정하지 않기, 교육용 예시임을 명시하기

## 5. AI 답변을 보고 내가 수정한 점

어려워서 쉽게 바꾼 표현: "Cyclic Redundancy Check" 같은 기술 용어를 "데이터 전송 오류를 알려주는 신호" 수준으로 풀어서 바꿨습니다.

실제 정보처럼 보일 수 있어 삭제하거나 수정한 표현: 예시에 등장할 수 있는 실제 장비명, IP 주소, 티켓 번호는 사용하지 않고 "SAMPLE-RACK-01"처럼 교육용임을 알 수 있는 표현으로 바꿨습니다.

너무 단정적이라 바꾼 표현: "Link Down의 원인은 케이블 문제입니다"처럼 원인을 하나로 단정하는 문장을, "케이블 문제일 수도, 장비 문제일 수도 있다"는 식으로 가능성을 열어두는 문장으로 바꿨습니다.

## 6. 오늘의 느낀 점

오늘 처음 알게 된 용어는: ToR Switch, Handover Memo처럼 실제 현장 흐름과 맞닿은 용어를 처음 접했습니다.

가장 헷갈렸던 용어는: SOP와 SLA가 비슷하게 들려서 처음에는 구분이 잘 되지 않았습니다.

AI 답변을 사용할 때 조심해야 할 점은: AI가 정리해준 설명이 실제 절차처럼 단정적으로 들릴 수 있어, 원인이나 기준을 하나로 확정 짓지 않았는지 직접 확인해야 한다고 느꼈습니다.

다음 차시에서 더 알고 싶은 내용은: SLA 기준 위반이 실제로 Escalation·Incident Report 작성으로 이어지는 흐름을 더 자세히 알아보고 싶습니다.
